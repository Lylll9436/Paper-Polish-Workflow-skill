**## Purpose**
One paragraph: Skill reads a full academic paper and verifies logical consistency across sections. It identifies four types of issues — argument chain gaps, unsupported claims, terminology inconsistencies, and number contradictions — and produces a two-part report. Part 1 maps each section's primary claim as an Argument Chain View table. Part 2 categorizes every identified issue with a problem description, impact statement, and one-sentence directional suggestion. The Skill never rewrites text; it identifies and suggests only.

**## Trigger**
Activates when user asks to check, verify, or find logical problems in their paper. Include bilingual examples (English + Chinese). Six example invocations covering: check the logic, verify argument chain, find logical inconsistencies, 检查论文逻辑, 验证论证链, 帮我找逻辑问题.

**## Modes**
Table with two rows:
- direct (default): single-pass full-paper analysis, no intermediate confirmation steps
- batch: NOT supported — cross-section logic verification requires full-paper context

**## References**
Required: none (Logic Skill loads no reference files — pure analysis task, no expression patterns needed, no anti-AI patterns loaded).
Leaf hints: none.
Note: No journal template loading — logic verification is journal-agnostic.

**## Ask Strategy**
Minimal Ask Strategy since this is a direct-mode single-pass Skill:
1. Input format: file path or pasted text? (skip if obvious from how user invoked Skill)
2. Section titles in paper (ask only if section structure is unclear from input)
Rules: in direct mode with sufficient input, proceed without pre-questions. Do not ask more than 1 question before starting.

**## Workflow**
Four steps:

Step 1 — Input Guard:
- Accept input as file (Read tool) or pasted text
- Check completeness: if input appears to be a single section only (single heading, < 500 words, or missing Methods/Results/Discussion markers), refuse with "Cross-section verification requires the full paper. Please provide the complete manuscript."
- Parse section structure: identify section boundaries and titles as they appear in the paper

Step 2 — Analyze Four Issue Types (run ALL before building the table):
The order of internal analysis is: run all four issue type checks first, then derive the Argument Chain View.

For each issue type:

**Argument Chain Gaps (AC-):**
- Map each section's primary research claim or contribution
- Check: Does Introduction claim get supported by Methods design?
- Check: Do Results address the Methods' stated measurements?
- Check: Does Discussion draw conclusions that follow from Results?
- Check: Does Conclusion match Discussion's stated findings?
- For each chain break, create Issue AC-N with section references

**Unsupported Claims (UC-):**
- Scan for assertive statements without evidence, citation, or data reference
- Focus on bold claims in Introduction and Discussion sections
- For each unsupported claim, create Issue UC-N quoting the exact claim text

**Terminology Inconsistencies (TI-):**
- Track all technical terms and model/method names introduced in Introduction
- Compare usage across all sections for variation (explicit renames, abbreviations introduced mid-paper, mixed use of formal and informal names)
- For each inconsistency, create Issue TI-N quoting ALL variant terms verbatim with their section locations

**Number Contradictions (NC-):**
- Extract quantitative values stated in Introduction (sample size, accuracy claims, improvement figures)
- Compare against values stated in Results and Conclusion
- For each discrepancy, create Issue NC-N quoting both values and their locations

Step 3 — Build Part 1 Argument Chain View:
- After AC- analysis is complete, build the table using analysis evidence
- Status = "Connected" only when no AC- issue spans that section pair
- Status = "Gap" when one or more AC- issues involve that section transition
- Use section titles exactly as they appear in the paper

Step 4 — Write Report:
- Assemble two-part report (see locked format in interfaces block)
- Report shows Part 1 table first, then Part 2 categorized issues
- If file input: derive output filename as {input_filename_without_ext}_logic.md; write using Write tool
- If pasted text input: present report in conversation
- State issue count summary: "Found N issues: X AC-, Y UC-, Z TI-, W NC-."

**## Output Contract**
Table with outputs:
- logic_report: Markdown file {input_filename}_logic.md (file input) or conversation output (pasted text)
Include the locked two-part structure from interfaces as a format reference. Note: Argument Chain View table is English only. Issue entries in Part 2 are bilingual (English + Chinese blockquote).

**## Edge Cases**
Table covering:
- Input is partial (single section or too short) → refuse with locked message: "Cross-section verification requires the full paper."
- Paper has no Discussion section (e.g., short paper with Results → Conclusion) → adapt chain view to actual sections present; note missing sections in table
- Paper uses Chinese section titles → use Chinese titles in table and section references as-is
- No issues found in a category → include the category heading in Part 2 with "No issues identified in this category."
- User asks Skill to rewrite identified issues → decline: "This Skill identifies issues and suggests directions only. Use the Polish Skill for rewriting."
- Numbers appear in non-quantitative context (e.g., figure labels, section numbers) → ignore; focus on empirical claims only
- Terminology inconsistency is intentional abbreviation with explicit definition → do not flag if abbreviation is defined on first use in Introduction

**## Fallbacks**
Table covering:
- Structured Interaction unavailable → proceed without pre-questions in direct mode; ask file-vs-pasted only via plain text if unclear
- Read tool fails (file not found/unreadable) → ask user to paste paper text instead
- Write tool fails → present logic report in conversation; advise user to save manually as _logic.md
- Paper is extremely long (> 20,000 words) → warn user that analysis may be approximate for very long manuscripts; offer to focus on specific section pairs if needed

**## Examples**
One minimal invocation: user provides a file path to a paper with a known terminology inconsistency. Show Ask Strategy exchange (minimal — direct mode proceeds). Show truncated output:
- Part 1 table with 5 rows showing one "Gap" on Introduction → Methods connection
- Part 2 showing Issue AC-1 with problem/why/suggestion + Chinese blockquote, and Issue TI-1 quoting two variant terms verbatim

Line budget: aim for 230-265 lines. The reference loading section can be very short (no references loaded).
